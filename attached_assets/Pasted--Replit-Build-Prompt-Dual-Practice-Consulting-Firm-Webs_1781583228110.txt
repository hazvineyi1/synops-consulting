## 1. What to build

Build a professional, fast, accessible **full-stack web application** for
`[FIRM_NAME]`, a boutique consulting firm with **two integrated practice areas**
under one roof and a **SaaS / digital-products** arm. The site should read as
credible to enterprise, healthcare, higher-ed, and **government procurement**
audiences. Prioritize trust, clarity, and conversion (book a call / request a
proposal).

Three things must be **fully functional at launch**:

1. **Marketing site + working contact form** with lead capture and a newsletter
   signup (see §5 Contact).
1. **A real Client Portal with authentication**: registration, login, sessions,
   and a protected dashboard (see §5 Client Portal and §2.5 Security).
1. **An embedded, working SaaS product demo** on the Platforms page that a
   visitor can actually interact with (see §5 Platforms & SaaS).

## 2. Tech stack

- **Full-stack app.** Use **Next.js + TypeScript** (App Router) so the marketing
  pages, API routes, and authenticated portal live in one deployable project,
  OR **React + Vite + an Express/Node API** if you prefer, either is fine, but
  it must deploy cleanly on Replit and support server-side auth + a database.
- **Tailwind CSS** for styling. No heavy UI kit; build clean custom components.
- **Database:** use Replit’s built-in **PostgreSQL** (or SQLite for a simpler
  start) with a typed query layer (Prisma or Drizzle). Define schemas for:
  `users`, `contact_submissions`, `newsletter_signups`, and `demo_sessions`.
- **Authentication (real):** email + password with **bcrypt-hashed** passwords,
  secure **HTTP-only session cookies**, CSRF protection on mutating requests,
  and server-side route protection for `/portal/*`. Replit Auth is acceptable if
  it’s simpler, but the portal must genuinely gate content behind login.
- **Contact form backend:** validate server-side, persist submissions, and email
  a notification to `[CONTACT_EMAIL]` (gate the email provider behind an env var;
  degrade gracefully to logging + success state if unconfigured).
- Put all secrets in environment variables. **No secrets in the repo.**
- Mobile-first, fully responsive, **WCAG 2.1 AA compliant** (hard req, see §7).

## 2.5 Security & compliance (read before building the portal)

- This portal is **launch-grade, not a system of record for regulated data.**
  Build it to be safe for general business use, but include a clear note in the
  README and on the portal that it must **not** store **PHI / patient data or
  other regulated health information** until a proper **HIPAA / security review**
  is completed. The healthcare practice deals with Medicaid/managed-care data.
  This must never live in this app without compliant infrastructure (BAA,
  encryption at rest, audit logging, access controls).
- Enforce: hashed passwords (never plaintext), session expiry, rate-limiting on
  auth endpoints, server-side input validation everywhere, and a honeypot +
  basic rate limit on public forms.
- Do **not** build password reset over email, payment handling, or document
  upload of sensitive files in this pass. Stub those as “coming soon” so we add
  them with proper review later.

  EdTech, not flashy, not corporate-stock-photo generic.

  clean sans for body (e.g. Inter) and a slightly distinctive sans or humanist
  serif for headings. Pick a real pairing and apply it consistently.

  tints) without fracturing the brand. One firm, two specialties.
- Use simple custom SVG/line iconography, not clip-art. No copyrighted logos.
- Include a tasteful “Veteran of $1B+ provider operations” / “PhD-level AI &

### Platforms & SaaS (`/platforms`)

- Framing: beyond advisory, the firm **builds**: adaptive learning platforms,
  internal AI agents/tools, dashboards, and SaaS products.
- Capability cards: custom web apps, AI-powered learning tools, operations
  dashboards & reporting, secure data workflows.
- **Embedded working demo (required, fully interactive).** Build an
  **“Adaptive Reading & Reasoning Demo”** that a visitor can use without signing
  in. The whole point is to show that **one engine adapts across every level**,
  so it reads as relevant to an elementary teacher, a high-school department, and
  a university program alike.
  - **Step 1, pick a level:** Elementary (grades 3 to 5), Secondary (grades 6 to
    12), or Higher Ed. The same skill, reading comprehension and evidence-based
    reasoning, is used at all three because it is taught and assessed across the
    full K-12 to graduate span. The chosen level sets passage complexity,
    vocabulary, and question depth (main idea, then inference, then evaluating an
    argument’s support).
  - **Step 2, short adaptive set (5 to 8 items):** each item shows a brief
    passage and a question. After each answer the engine **adapts**: answer well
    and the next passage gets harder (longer text, deeper inference); miss one
    and the learner gets a scaffolded hint and an easier item. Track a running
    mastery estimate.
  - **Step 3, results screen:** estimated mastery, the recommended “next rung”
    (and an invitation to try a different level), and a small chart of the path
    taken. Persist anonymous runs to `demo_sessions`.
  - Build it **rules-based and self-contained** with a seeded item bank **per
    level** so it works with no API keys. Add an **optional** hook (behind an env
    var) to call an LLM for dynamic passage and hint generation; if the key is
    absent, fall back to the built-in bank. Do not hard-require any external AI
    service.
  - Keep all passages original and neutral (science, civics, everyday topics) so
    nothing is age-inappropriate at the lower band. This demo showcases the
    EdTech/AI practice and doubles as the “Request a demo” conversion path.
